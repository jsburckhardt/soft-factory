# CORE-COMPONENT-0002: Commit Standards

## Status

Adopted

## Purpose

Establish a consistent, machine-readable commit message format across all repositories using Soft Factory. Consistent commits enable automated changelog generation, semantic versioning, and clear project history.

## Scope

All git commits in any repository using the Soft Factory template. This applies to both human contributors and AI agents.

## Definition

### Rules
- All commit messages MUST follow the [Conventional Commits v1.0.0](https://www.conventionalcommits.org/en/v1.0.0/) specification.
- The commit message format is: `<type>[optional scope]: <description>`
- The type MUST be one of: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.
- The description MUST be lowercase, imperative mood, and no period at the end.
- The scope SHOULD identify the area of the codebase affected (e.g., `auth`, `api`, `docs`).
- Breaking changes MUST include `BREAKING CHANGE:` in the commit footer or `!` after the type/scope.
- AI-authored commits MUST include a `Co-authored-by` trailer.

### Interfaces
- Commit messages are the interface â€” no library or API is involved.
- PR titles MUST also follow Conventional Commits format.

### Expectations
- Every commit on a feature branch follows this format.
- The verifier agent enforces this before pushing.
- Squash merges use the PR title as the commit message.

## Rationale

Conventional Commits is the most widely adopted structured commit format. It integrates with semantic-release, standard-version, and changelog generators. It is language-agnostic and simple enough for both humans and AI agents to follow consistently.

## Usage Examples

```
feat(auth): add JWT token refresh endpoint

fix: resolve null pointer on empty user input

docs: update CONTRIBUTING.md with RPIV pipeline

refactor(api)!: rename /users endpoint to /accounts

BREAKING CHANGE: /users endpoint no longer exists, use /accounts instead.

chore: upgrade APS skill to v1.2.2

Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>
```

## Integration Guidelines

- The verifier agent checks commit format before pushing to remote.
- PR titles follow the same format for squash-merge consistency.
- Use `feat` for new functionality visible to users.
- Use `fix` for bug fixes.
- Use `docs` for documentation-only changes.
- Use `refactor` for code changes that neither fix bugs nor add features.
- Use `chore` for maintenance tasks (dependency updates, config changes).

## Exceptions

- Initial repository setup commits (e.g., `git init`) are exempt.
- Merge commits generated by GitHub are exempt.
